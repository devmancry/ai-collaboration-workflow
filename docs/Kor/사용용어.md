

# 1. 사용하는 용어

AI 협업 문서에서는 동일한 개념에 항상 동일한 용어를 사용한다.

## 1.1. 용어 정리의 목적

AI 협업에서 가장 중요한 원칙 중 하나는 **용어의 일관성**이다.

같은 의미를 가진 단어가 여러 개 존재할 경우 사람에게는 동일하게 보일 수 있지만, AI는 이를 서로 다른 개념으로 해석할 가능성이 있다. 이러한 문제는 설계, 문서 작성, 코드 구현으로 이어지는 전체 작업 과정에서 혼란을 발생시킬 수 있다.

또한 이 프로젝트의 작업은 **설계 → 문서 → 구현 → 코드**로 이어지는 흐름을 가지기 때문에, 실제 프로그래밍과 소프트웨어 개발에서 사용되는 용어를 사용하는 것이 유리하다.

따라서 이 문서를 포함한 여러 문서에서는 **영어 용어와 한글 설명이 함께 사용될 수 있다.** 이는 표현의 혼용이 아니라 **작업의 명확성과 용어 일관성을 유지하기 위한 의도적인 방식**이다.

아래의 용어 정리는 새로운 단어를 만들기 위한 것이 아니라, **상황이나 분야에 따라 다르게 해석될 수 있는 용어의 의미를 명확하게 정의하기 위한 것**이다.

---

## 1.2. 용어 정의의 범위

이 문서는 모든 단어를 설명하기 위한 사전이 아니다.

다음과 같은 경우에만 용어 정의를 제공한다.

- 이 프로젝트에서 **특정한 의미로 사용되는 단어**
- 일반적인 의미와 **다르게 사용될 수 있는 단어**
- 설계, 문서, 코드 전반에서 **핵심 개념으로 반복 사용되는 단어**
- 상황이나 분야에 따라 **해석이 달라질 수 있는 단어**

다음과 같은 단어는 별도의 설명 없이 일반적인 의미로 사용된다.

- 프로그래밍 및 소프트웨어 개발에서 널리 사용되는 일반 용어
    
    (예: `Tag`, `Operator`, `Validation`, `Module` 등)
    

이러한 용어는 일반적인 소프트웨어 개발 문맥에서 사용되는 의미를 따른다

---

## 1.3. 용어 정리

줄여 쓰거나 한글과 영어를 함께 사용하는 단어는 괄호 안에 함께 표기한다.

괄호 안에 두 개 이상의 표현이 들어갈 경우에는 **“/”**로 구분한다.

두 개 이상의 표현이 있을 경우 **사용 빈도가 높은 표현을 앞에 배치한다.**

---

## 1.4. 용어 목록

### 시스템 (System)

여러 구성 요소가 함께 동작하여 **하나의 목적을 수행하는 구조 전체**를 의미한다.

주요 특징

- 여러 모듈과 기능이 함께 구성됨
- 내부 구성 요소 간의 관계가 존재함
- 특정 목적을 수행하는 구조를 형성함

예

```
Combat System
Enemy AI System
```

---

### 철학 (Philosophy)

시스템 설계에서 **일관되게 유지되어야 하는 설계 방향과 사고 방식**을 의미한다.
철학은 구체적인 규칙이나 구현 방식보다 **상위 개념의 설계 기준**으로 사용된다.

예

```
UI는 시스템 상태를 보여주는 창이며 가능한 한 직접적인 상태 변경을 수행하지 않는다
게임 로직은 데이터 중심 구조를 우선한다
Player와 AI는 동일한 게임 규칙을 사용한다
```

---

### pipeline (파이프라인)

여러 단계의 작업이 **정해진 순서로 연결되어 실행되는 구조**를 의미한다.

Pipeline은 일반적으로 **입력 → 처리 단계 → 결과**의 흐름으로 구성된다.

예

```
Player Input
→ Action Selection
→ Action Chain
→ Result
```

---

### 규칙 (Rule)

시스템이 동작할 때 **반드시 지켜야 하는 동작 규칙**을 의미한다.

규칙은 시스템의 안정성과 예측 가능한 동작을 유지하기 위해 정의된다.

예

```
Asset Load는 Operator에서만 수행된다
Validation은 Tag 기반 시스템으로 실행된다
Unit 이동은 Grid 기반 Pathfinding을 사용한다
```

---

### System Contract

시스템 또는 모듈이 **반드시 지켜야 하는 약속과 보장 조건**을 의미한다다.

다른 시스템이 해당 시스템을 사용할 때 **신뢰할 수 있는 동작을 보장하는 규칙**이다.

예

```
Gravity Validation은 항상 현재 Scene 상태를 기준으로 실행된다
Action Chain이 시작되면 반드시 End 단계까지 실행된다
Asset Load가 성공하면 Scene에 Asset Instance가 존재한다
```

---

### Architecture

시스템의 **구조와 구성 요소 사이의 관계**를 정의한다.

주요 내용

```
이 시스템은 어떤 구성 요소로 이루어져 있는가?
각 구성 요소는 어떻게 연결되는가?
전체 구조는 어떤 원칙으로 작동하는가?
```

포함되는 것

- 시스템 구성 요소
- 모듈 관계
- 데이터 흐름
- 시스템 철학
- 불변 규칙 (invariants)

포함되지 않는 것

- 함수 단위 규칙
- 구체적인 구현 방식
- API 상세

---

### Spec (Specification / 스펙)

특정 기능이나 시스템을 **구현하기 위한 상세 규격**을 의미한다.

Spec은 기능이 **어떻게 동작해야 하는지를 정확하게 정의**한다.

주요 내용

- 동작 규칙
- 입력 / 출력
- 상태 변화
- 구현 요구사항

---

### **Source of Truth**

특정 개념, 규칙, 또는 시스템에 대해 **가장 신뢰해야 하는 기준 문서 또는 기준 정보**를 의미한다.

Source of Truth는 해당 내용에 대해 **최종적인 기준이 되는 위치**를 정의하며,

다른 문서나 설명은 이 기준을 **참조하거나 보조 설명을 제공하는 역할**을 한다.

이 개념을 명확히 정의함으로써 다음과 같은 문제를 방지할 수 있다.

- 동일한 규칙이 여러 문서에서 서로 다르게 설명되는 문제
- 구현과 설계 문서 사이의 불일치
- 문서 간 책임 영역의 혼동

따라서 각 문서 유형은 **자신이 담당하는 영역의 Source of Truth 역할을 가진다.**

예

```
Core 문서 → Architecture Source of Truth
Spec 문서 → Rule Source of Truth
Reference 문서 → Implementation Source of Truth
```
