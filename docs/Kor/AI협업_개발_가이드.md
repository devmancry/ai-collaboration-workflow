# AI와 협업하는 개발 방법

---

# 1. 소개

최근 AI를 활용한 개발 방식이 빠르게 늘어나고 있다.

그러나 많은 경우 다음과 같은 두 가지 한계가 있다.

- AI에게 대부분의 작업을 맡기면서 **프로젝트의 구조와 방향을 통제하기 어려워지는 문제**
- AI를 단순 보조 도구로만 사용하여 **작은 작업이나 반복 작업에만 활용하는 경우**

이 문서는 이러한 한계를 보완하고
**AI의 장점을 적극적으로 활용하여 설계와 구현을 진행하는 협업 방식**을 설명한다.

## 1.1. 핵심 개념

```
AI를 코드 생성 도구가 아니라
설계와 구현을 함께 수행하는 협업 파트너로 사용한다.
```

이 방식의 특징은 다음과 같다.

- 사람이 **설계의 주도권을 유지한다**
- AI를 **설계와 구현 과정에서 리뷰어이자 협업 파트너, 동시에 학습 도구로 적극 활용한다**
- 작업은 **설계 → 문서 → 구현 → 검토 → 문서 업데이트** 흐름으로 진행된다
- 문서를 중심으로 협업이 이루어지기 때문에 **대규모 프로젝트에서도 안정적으로 작업을 이어갈 수 있다**

이 문서를 읽는 독자는 다음 두 가지를 얻을 수 있다.

- 이 협업 방식의 구조와 원리를 이해한다
- 실제 프로젝트에서 **같은 방식으로 작업을 시작할 수 있다**

---

## 1.2. 기본 원칙

이 방법의 핵심은 **AI 협업 구조를 명확하게 분리하는 것**이다.

많은 AI 기반 개발 방식은 다음과 같은 단순한 협업 구조를 가진다.

```
Human
  ↓
AI
```

하지만 이 방법에서는 협업 구조를 **세 가지 역할로 나눈다.**

```
Human (기획자)
Design AI (설계 AI)
Implementation AI (코딩 AI)
```

각 역할은 개발 과정의 서로 다른 부분에 집중한다.

```
Human
→ 설계 방향 결정 및 최종 판단

Design AI
→ 설계 논의, 구조 검토, 문서 정리

Implementation AI
→ 구현 계획 작성, 코드 생성, 테스트 제안
```

**실제 설계**는 사람이 주도권을 가지고 Design AI와 협업한다.

**실제 구현**은 Implementation AI가 코드를 생성하고 사람과 Design AI가 코드를 리뷰한다.

즉, 이 협업 구조는 실제 개발 과정에서 다음과 같은 흐름으로 진행된다.

```
아이디어
(Human)
     │
     ▼
설계 논의
(Human + Design AI)
     │
     ▼
문서 정리
(Human + Design AI)
     │
     ▼
코드 생성
(Implementation AI)
     │
     ▼
테스트
(Human)
     │
     ▼
코드 검토 및 피드백
(Human + Design AI)
```

이처럼 협업 구조를 분리하면

설계와 구현 과정의 역할이 명확해지고 AI 협업을 보다 안정적으로 진행할 수 있다.

---

## 1.3. 이 방법이 중요한 이유

소프트웨어 개발에서 **설계는 매우 중요한 단계**이다.

시스템의 구조와 규칙은 설계 단계에서 결정되며,
이후 구현의 안정성과 확장성에도 큰 영향을 미친다.

AI를 활용한 개발에서도 이 점은 동일하다.

코드를 생성하는 능력만큼이나 **설계를 충분히 논의하고 검토하는 과정**이 중요하다.

그러나 프로그래머가 아닌 사람에게 설계를 고민하는 것은 쉽지 않다.

어떤 구조가 좋은지, 어떤 문제가 발생할 수 있는지,
어떤 접근 방식이 더 적절한지 판단하기 어렵기 때문이다.

AI는 이 과정에서 단순한 코드 생성 도구가 아니라
**설계 논의를 함께 진행할 수 있는 협업 파트너이자 학습 도구**로 활용될 수 있다.

AI와의 대화를 통해

- 모르는 개념을 질문하고 이해할 수 있으며
- 더 나은 구조와 접근 방식을 탐색하고
- 예상되는 문제를 미리 발견하며
- 새로운 아이디어와 영감을 얻을 수 있다.

이러한 과정은 특히 **비프로그래머가 설계를 이해하고 발전시키는 데 큰 도움이 된다.**

이 방법에서는 이러한 설계 논의와 코드 생성을 **서로 다른 역할로 분리한다.**

- **Design AI**는 설계 논의와 구조 검토에 집중하고
- **Implementation AI**는 구현과 코드 생성에 집중한다.

이러한 역할 분리를 통해 설계 안정성과 구현 효율을 동시에 유지할 수 있다.

---

# 2. 문서의 종류와 역할

이 협업 구조에서 **문서는 매우 중요한 역할을 한다.**

각 문서는 서로 다른 책임 영역을 가지며,
단순히 AI에게 정보를 전달하기 위한 파일이 아니라
**설계 논의와 구현을 연결하는 협업 구조의 일부로 사용된다.**

## 2.1. 사용하는 문서

이 방법에서는 다음과 같은 문서 구조를 사용한다.

```
Design 문서
→ 사람이 의도한 시스템 설계

Core 문서
→ AI와 합의된 시스템 구조

Spec 문서
→ AI를 위한 구현 규칙

Reference 문서
→ 실제 코드 구조 설명
```

이러한 구조를 통해

- 설계 의도가 명확하게 기록되고
- AI가 일관된 기준을 유지하며 작업할 수 있으며
- 긴 개발 과정에서도 구조를 안정적으로 유지할 수 있다.

---

## 2.2. 문서 작성 방식

이 협업 구조에서는 문서의 종류에 따라 작성 방식이 다르다.

Design 문서는 사람이 중심이 되어 작성하며, 
Core, Spec, Reference 문서는 AI가 초안을 작성하고 사람이 검토하는 방식으로 진행된다.

```
Design
→ 사람이 중심이 되어 작성 (AI는 보조 역할)

Core
→ AI 초안 작성 + 사람 검토

Spec
→ AI 초안 작성 + 사람 검토

Reference
→ AI 초안 작성 + 사람 검토
```

---

## 2.3. Design

**Design 문서**는 사람 중심의 설계 문서이다.

이 문서는 일반적인 실무에서 사용되는 **기획 문서 또는 설계 문서**에 해당한다.

예를 들어 게임 프로젝트에서는 **게임 기획서**와 같은 문서가 된다.

Design 문서는 시스템을 이해하기 위한 설명을 중심으로 작성되며,
다음과 같은 내용을 포함할 수 있다.

```
프로젝트 또는 시스템의 목적
핵심 아이디어와 설계 철학
전체 구조와 주요 시스템 개념
주요 기능이나 시스템의 설명
설계 의도와 문제 해결 접근 방식
향후 확장 방향이나 고려 사항
```

Design 문서는 구현 규칙이나 API 같은 세부 내용보다는
**시스템의 개념과 설계 방향을 설명하는 데 집중한다.**

즉 Design 문서는 **사람이 시스템을 이해하기 위한 문서**이며
프로젝트 설계의 전체 방향을 설명하는 **설계 기준 문서(Project Design Source of Truth)**로 사용된다.

---

## 2.4. Core

**Core 문서**는 시스템의 전체 구조와 핵심 규칙을 설명한다.

Core 문서는 설계 개념을 **구조적인 형태로 정리한 문서**이며
시스템이 어떤 구조로 구성되는지를 명확하게 정의한다.

여기에는 다음과 같은 내용이 포함된다.

- 시스템 구성 요소
- 시스템 간 관계
- 데이터 흐름
- 핵심 규칙 (System Contract)

Core 문서는 구현 여부와 관계없이 **전체 시스템 구조를 포함하며**
시스템 구조를 정의하는 **아키텍처 기준 문서(Architecture Source of Truth)**로 사용된다.

---

## 2.5 Spec

**Spec 문서**는 구현 규칙을 정의하는 문서이다.

이 문서는 특정 기능이나 시스템이 **어떻게 동작해야 하는지**를 구체적으로 설명한다.

예를 들어 다음과 같은 내용이 포함된다.

- 동작 규칙
- 입력 / 출력
- 상태 변화
- 구현 요구사항

Spec 문서는 구현 대상의 상태도 함께 기록한다.

```
STATUS
implemented
partial
planned
```

Spec 문서는 시스템 동작 규칙을 정의하는 **구현 규칙 기준 문서(Rule Source of Truth)**로 사용된다.

---

## 2.6. Reference

**Reference 문서**는 실제 구현 구조를 설명한다.

이 문서는 현재 코드 상태를 기준으로 작성되며

- 코드 구조
- 처리 흐름
- 주요 API

등을 설명한다.

Reference 문서는 실제 구현 구조를 설명하는 
**구현 구조 기준 문서(Implementation Source of Truth)**로 사용된다.

따라서 **미구현 내용은 Core와 Spec 문서에만 존재해야 한다.**

---

# 3. 개발 진행 순서

이 협업 구조에서 개발은 다음과 같은 **반복 사이클**로 진행된다.

```
아이디어
→ 설계 논의
→ 문서 정리
→ 구현
→ 검토
→ 문서 최신화
```

각 단계는 서로 다른 역할을 가지며 
이 사이클을 반복하면서 시스템 설계와 구현이 점점 정교해진다.

---

## 3.1. 아이디어 단계

개발은 **사람이 제시하는 아이디어**에서 시작된다.

사람이 구체적인 아이디어의 초안을 제시하고, 
AI에게 이를 설명하여 함께 논의하며 아이디어를 구체화한다.

이 단계에서는 다음과 같은 내용이 정리될 수 있다.

- 해결하려는 문제
- 구현하려는 기능
- 시스템 확장 방향
- 초기 설계 아이디어

---

## 3.2. 설계 논의 단계

아이디어가 정리되면 **설계 논의 단계**가 진행된다.

이 단계에서는 사람과 **Design AI**가 함께 
개발자가 실제로 구현할 수 있을 정도로 **구체적인 시스템 구조와 규칙**을 논의한다.

이 과정에서 중요한 점은 **사람이 설계의 주도권을 유지하는 것**이다.

사람은 Design AI에게 다음과 같은 질문을 적극적으로 던져야 한다.

- 예상되는 문제는 무엇인가
- 더 나은 구조나 접근 방식은 없는가
- 새로운 개념이나 기술이 필요한가

또한 AI가 제안한 내용을 그대로 받아들이기보다는 
사람이 스스로 판단하고 **설계 방향을 결정해야 한다.**

이 단계에서는 **코드를 먼저 작성하지 않는다.**

설계 논의를 통해 시스템 구조와 규칙을 충분히 검토한 후

다음 단계로 진행한다.

---

## 3.3. 문서 정리 단계

설계 논의에서 결정된 내용은 **문서로 정리된다.**

이 단계에서는 다음 순서로 문서가 정리된다.

```
Design → Core → Spec
```

### 3.3.1. Design 정리

먼저 **Design 문서가 정리된다.**

Design 문서는 사람이 중심이 되어 작성하며 추상적인 수준에서 설계 방향을 설명한다.

---

## 3.4. 구현 단계

문서 정리가 완료되면 **구현 단계**로 넘어간다.

이 단계에서는 **Implementation AI**가 실제 구현 작업을 수행한다.

Implementation AI는 다음 문서를 기준으로 작업한다.

```
Core
Spec
```

Core 문서는 시스템의 전체 구조를 제공하며 
Spec 문서는 구현해야 할 기능의 **구체적인 규칙**을 제공한다.

---

### 3.4.1. 구현 계획 작성

이 단계에서 중요한 점은 **AI가 바로 코드를 생성하지 않는다는 것**이다.

먼저 Implementation AI가 **구현 계획(Implementation Plan)**을 작성한다.

이 계획에는 다음과 같은 내용이 포함될 수 있다.

- 구현 대상 기능의 구조
- 필요한 클래스 또는 모듈
- 데이터 구조
- 처리 흐름
- 예상되는 구현 단계

---

### 3.4.2. 구현 계획 검토

작성된 구현 계획은 **사람과 Design AI가 함께 검토한다.**

이 단계에서는 다음을 확인한다.

- 계획이 Core 구조와 일치하는지
- Spec 규칙을 정확히 반영하고 있는지
- 설계 의도와 충돌하는 부분이 없는지
- 더 나은 구조가 존재하지 않는지

필요할 경우 이 단계에서 계획이 수정된다.

---

### 3.4.3. 코드 생성

구현 계획이 **승인된 이후에만** Implementation AI가 실제 코드를 생성한다.

즉 구현 과정은 다음과 같은 순서로 진행된다.

```
Spec 정의
→ Implementation Plan 작성
→ Plan 검토
→ Plan 승인
→ 코드 생성
```

이 구조를 통해

- 설계와 구현 사이의 불일치를 줄이고
- 코드 품질을 높이며
- AI 협업의 안정성을 유지할 수 있다.

---

## 3.5 검토 단계

구현이 진행된 후에는 **검토 단계**가 수행된다.

이 단계에서는 다음을 확인한다.

- 코드가 Spec 규칙을 따르는지
- 시스템 구조가 Core와 일치하는지
- 예상하지 못한 문제가 발생하지 않았는지

필요할 경우 Design AI를 다시 사용하여 구조나 규칙을 재검토할 수 있다.

---

## 3.6. 문서 최신화 단계

마지막 단계에서는 문서를 **현재 상태에 맞게 업데이트한다.**

이 단계에서 업데이트되는 문서는 다음과 같다.

```
Spec
Reference
```

- Spec → 구현 상태 업데이트
- Reference → 실제 코드 구조 설명

이 과정을 통해 문서는 항상 **현재 프로젝트 상태와 동기화**된다.

---

## 3.7. 개발 반복 사이클

이 사이클은 한 번으로 끝나는 과정이 아니라

프로젝트 진행 동안 계속 반복된다.

```
아이디어
→ 설계 논의
→ 문서 정리
→ 구현
→ 검토
→ 문서 최신화
→ 아이디어...
```

이 반복 과정을 통해

- 설계는 점점 정교해지고
- 문서는 프로젝트의 지식 기반으로 발전하며
- AI 협업은 점점 더 안정적으로 작동하게 된다.

# 4. 진행 팁

위의 방법과 순서대로 진행할 때, 다음과 같은 타이밍에서 대화를 정리하고 새 대화로 넘어가는 것이 좋다.

- 하나의 개발 사이클이 완료되었을 때
- 하나의 설계 또는 구현 단위가 완료되었을 때
- 대화가 길어져 AI가 맥락을 놓치거나 성능 저하가 나타나기 시작할 때

또한 개발 사이클이 하나 완료될 때마다 버전이 일치하지 않는 구버전 문서가 남아 있지 않도록 해야 한다.

문서를 업데이트할 때에는 정보가 실수로 누락되거나 현재 상태와 어긋나지 않도록 꼼꼼히 확인하는 것이 중요하다.
