# AI 협업 사례 - 설계 공백을 발견하고 Spec을 작성한 과정

이 문서는 **AI와 협업하는 개발 방법** 문서에서 설명한 방식이 
실제 설계 과정에서 어떻게 적용되는지를 보여주는 사례 기록이다.

이 사례는 **완전히 새로운 시스템을 처음부터 설계한 과정이 아니라**, 
이미 존재하는 설계 문서를 기반으로 **구현을 준비하는 과정에서 설계 공백을 발견하고 보완한 사례**이다.

프로젝트에는 이미 다음과 같은 문서들이 존재하고 있었다.

```
Design
Core
일부 기능 Spec
일부 Reference
```

이 상태에서 **다음 구현 작업을 시작하기 위해 AI와 대화를 시작했고**, 
대화 과정에서 **기존 설계에 부족한 부분이 발견되었다.**

그 결과

- Design에 없던 개념이 추가 설계되고
- 시스템 구조가 정리되었으며
- 해당 내용을 기반으로 **Spec 문서가 작성되었다.**

이 문서는 그 과정의 흐름을 기록한 것이다.

---

## 0. 사례 흐름 요약

이 사례에서 진행된 과정은 다음과 같다.

```
구현 목표 설정
→ 설계 질문 발생
→ 기존 설계의 공백 발견
→ 유닛 구조 설계
→ Core 문서 정리
→ Spec 작성
```

이 사례는 **AI와 협업하여 설계 공백을 발견하고 보완하는 과정**을 보여준다.

---

## 1. 구현 목표에서 대화가 시작된다

이 대화는 하나의 구현 작업을 준비하는 과정에서 시작되었다.

목표는 **게임의 승패 조건 시스템을 구현하는 것**이었다.

승패 조건을 만들기로 하고 나서 가장 먼저 이런 질문을 했다.

- **Protagonist를 어떻게 구분할 것인가?**
- **MotherAI를 어떻게 구분할 것인가?**

예를 들어

- 주인공이 죽으면 패배인가?
- 마더 AI가 파괴되면 패배인가?

같은 규칙을 구현하려면 먼저 **그 유닛들이 무엇인지 구분할 수 있어야 하기 때문**이다.

이 질문은 단순한 구현 질문이 아니라 **시스템 구조와 관련된 설계 질문**이었다.

```
[방법론 단계 — 설계 논의 단계]
```

---

## 2. AI의 제안을 검토하며 질문을 이어간다

AI는 몇 가지 방법을 제안한다.

예를 들어

- 역할(Role)로 구분하는 방법
- 특정 ID로 구분하는 방법
- 태그(Tag)를 사용하는 방법

하지만 여기서 바로 결론을 내리지 않는다.

대신 다음과 같은 질문이 이어진다.

- 같은 캐릭터가 여러 번 등장할 수 있다면?
- 적이 나중에 아군이 된다면?
- 스테이지마다 능력치가 다르면?
- 복장이 바뀌는 경우는 어떻게 관리할까?

이 질문들은 단순한 반대 의견이 아니라

**설계가 깨질 수 있는 상황을 미리 점검하는 과정**이다.

AI 협업에서는 AI의 제안을 그대로 받아들이기보다는 
이처럼 **경계 상황을 확인하면서 설계를 검증하는 과정**이 중요하다.

```
[방법론 단계 — 설계 논의 단계]
```

---

## 3. 대화의 초점이 유닛 구조 설계로 이동한다

이 질문들을 통해 점점 문제가 명확해진다.

단순히

```
주인공을 구분하는 방법
```

이 필요한 것이 아니라

```
게임 전체에서 유닛을 어떻게 관리할 것인가
```

라는 **더 큰 구조 설계 문제**가 존재한다는 점이다.

그래서 다음과 같은 개념들이 등장한다.

- 원본 유닛
- 플레이어가 보유한 유닛
- 스테이지에서 배치되는 유닛
- visual variant (외형 변경)

이 시점부터 대화는 사실상 **유닛 데이터 구조 설계**로 넘어간다.

이 설계는 기존 Design 문서에는 존재하지 않았던 내용이었다.

즉, 구현 준비 과정에서 **Design의 설계 공백이 발견된 것**이다.

---

## 4. 인간이 설계 초안을 제시한다

여기서 중요한 전환이 하나 일어난다.

AI가 설계를 만들어주는 것이 아니라 
**사용자가 직접 구조 초안을 제시한다.**

예를 들어 다음과 같은 구조가 제안된다.

```
원본 유닛 DB
플레이어 유닛 리스트
스테이지 배치 데이터
visual variant
```

이 구조를 바탕으로 다시 질문이 이어진다.

- 같은 유닛을 여러 번 스폰할 수 있는가?
- 적 버전과 아군 버전을 분리할 필요가 있는가?
- 외형만 다른 경우는 어떻게 처리할까?

이 과정에서 설계가 점점 구체화된다.

AI 협업에서 중요한 점은 다음과 같다.

```
AI가 설계를 대신 만드는 것이 아니라
사람이 구조 초안을 제시하고
AI와 함께 설계를 검토한다
```

---

## 5. 설계를 문서로 정리한다

구조가 어느 정도 정리되자

다음 단계로 **문서 작성 단계**가 진행된다.

```
[방법론 단계 — 문서 정리 단계]
```

먼저 설계 내용을 **Core 수준의 구조 문서**로 정리한다.

여기에는 다음과 같은 내용이 포함된다.

- 유닛 데이터 구조
- 각 데이터의 역할
- override 규칙
- visual variant의 역할
- 적 ↔ 아군 전환 시 연속성 처리

이 문서는 **사람과 AI가 읽고 이해할 수 있는 구조 문서**로 작성된다.

---

## 6. Spec 문서를 작성한다

Core 수준의 구조가 정리된 다음에는

**구현 규칙을 정의하는 Spec 문서**가 작성된다.

Spec 문서는 특히 다음을 명확히 한다.

1. 무엇을 구현해야 하는가
2. 무엇은 아직 구현하지 않는가
3. 어떤 규칙을 반드시 지켜야 하는가

Spec 문서는 **Implementation AI가 코드를 생성할 때 사용하는 기준 문서**가 된다.

---

## 7. 구현을 한 번에 진행하지 않기로 결정한다

마지막으로 중요한 결정이 하나 내려진다.

이 시스템을 **한 번에 구현하지 않기로 한 것**이다.

대신 구현 단계를 나누기로 한다.

예를 들어 다음과 같은 방식이다.

```
Phase 1 — 유닛 스폰 기반 구조
Phase 2 — 플레이어 유닛
Phase 3 — 능력치 override
Phase 4 — visual variant
```

이렇게 구현 단계를 나누면

- AI가 과도하게 확장하거나
- 여러 시스템을 한 번에 연결해 버리는 문제를

줄일 수 있다.

---

## 8. 이 사례의 현재 단계

이 사례는 **Spec 문서를 작성하는 단계에서 마무리되었다.**

방법론 기준으로 보면 현재 상태는 다음과 같다.

```
Design
→ Core
→ Spec
```

다음 단계에서는 다음 작업이 진행될 예정이다.

```
Design 업데이트
Spec 기반 Implementation Plan 작성
Implementation Plan 검토
코드 생성
```

Implementation Plan과 코드 생성 단계의 협업 과정은 다음 사례 문서에서 다룰 예정이다.

---

# 이 사례가 보여주는 AI 협업 방식

이 사례는 AI를 단순한 코드 생성 도구로 사용하는 방식과 다르다.

이 과정은 다음 특징을 가진다.

```
구현 목표에서 시작한다
→ 설계 질문을 통해 문제를 발견한다
→ 인간이 구조 초안을 제시한다
→ AI와 함께 설계를 검토한다
→ 구조를 Core 문서로 정리한다
→ 구현 규칙을 Spec으로 정의한다
```

즉,

```
AI에게 코드를 요청하기 전에
설계를 충분히 정리하는 방식
```

이다.

또한 이 사례는 **구현 준비 과정에서 설계 공백을 발견하고 보완하는 AI 협업 방식**을 보여주는 실제 예시이기도 하다.

# 이 사례를 따라 하는 방법

이 사례와 같은 방식으로 AI와 설계 논의를 시작하려면 
다음과 같은 질문에서 시작할 수 있다.

```
1 구현하려는 기능을 먼저 명확히 한다

예
"승패 조건 시스템을 구현하려고 한다."

2 해당 기능이 의존하는 개념을 질문한다

예
"주인공을 시스템적으로 어떻게 구분할 수 있을까?"

3 설계가 깨질 수 있는 상황을 질문한다

예
"같은 캐릭터가 여러 번 등장할 수 있다면?"

4 구조 초안을 제시하고 검토한다

예
"원본 유닛 DB / 플레이어 유닛 / 스테이지 유닛 구조를 생각 중인데 어떨까?"
```

이 과정을 통해 **설계 질문 → 구조 설계 → Core → Spec** 흐름을 만들 수 있다.
